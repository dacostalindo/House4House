## For future Claude

This is the **full extraction** working document for the Aveiro PDM Regulamento, broken down into atomic constraints. Each row in the tables below maps 1:1 to a row in `gold_analytics.dim_pdm_constraint` (314 rows materialized via `/tmp/md_to_sql.py`). Once finalized, this can move to status `verified`. **As of 2026-06-04 — uncommitted, branch `feature/dim-pdm-constraint-aveiro`. See [[log]] § 2026-06-04 for full context-save before /compact.**

**Tables here use either 9 columns (legacy) or 11 columns (new):** columns 8-9 of the 11-col format are `applies_to_zone_types` (TEXT[], comma-separated inside backticks) + `applies_when_land_classification` (TEXT, "Solo Rústico" or "Solo Urbano"). These filter overlay-conditional rules to specific SRUP zone_types + land_classifications.

**Zone_pattern is polymorphic**: matches either (a) a `subcategoria` of `silver_geo.zoning` (exact string), (b) umbrella slug (ALL_PDM / ALL_SOLO_URBANO / ALL_SOLO_RUSTICO), or (c) a SRUP layer `constraint_code` (RAN, ZPE, Perigosidade_Incendio_Rural, etc.). The SRUP-overlay matching happens at the `silver_regulatory.srup_constraints` level, NOT in silver_geo.zoning (zoning is too coarse).

**Iteration discipline:** every `source_text` cell is verbatim from the PDM Regulamento — no paraphrasing, no summary, no truncation. Cells are wrapped in `` ` `` backticks so the literal `|` (used in `carve_out` slug separators) is escaped as `\|`. Cells can wrap to multiple visual lines; the underlying data is one logical row.

## Resolution at query time (fn_assess_polygon)

For a given drawn polygon, the resolver returns constraints from:

1. **Pattern 4 zones** — ILIKE match against `silver_geo.zoning.land_designation` where polygon ∩ zoning
2. **Pattern 1 umbrellas** — all rows where `zone_pattern = 'ALL_PDM'`; plus `'ALL_SOLO_URBANO'` if polygon ∩ Solo Urbano zoning; plus `'ALL_SOLO_RUSTICO'` if polygon ∩ Solo Rústico zoning
3. **Pattern 2 overlays** — for each overlay slug, JOIN with the corresponding SRUP/PDM layer in silver if the polygon intersects (e.g. EEM via Planta da Estrutura Ecológica, POC_OMG_FAIXA_* via Planta Faixas, PORNDSJ via Planta Condicionantes, IC via Anexo A list, etc.)
4. **Pattern 3 UOPGs** — if polygon falls in an UOPG boundary (planta de ordenamento UOPGs), JOIN those rows too

The overlay layers (Pattern 2) require **per-overlay spatial datasets** in silver — these mostly already exist as SRUP silver tables (`silver_regulatory.srup_*`) but `EEM`, `POC_OMG_*`, `PORNDSJ_*`, `ARPSI_Floodplain_*`, `IC`, `PATRIMONIO_ARQUEOLOGICO`, `ESPACOS_CANAIS`, `EQUIPAMENTOS_EDUCATIVOS`, `ZONAMENTO_ACUSTICO_*` will need to be ingested separately (Planta de Ordenamento subplots from the PDM publication).

## Zone-pattern ↔ SRUP cross-reference (deep PDM audit, 2026-06-04)

Maps each PDM zone_pattern to the SRUP layer(s) it textually names or definitionally encompasses. Read by the Python generator into the auto-generated `cross_refs_srup TEXT[]` column on `dim_pdm_constraint`. Then `silver_regulatory.srup_constraints` matches PDM rules to SRUP features via `srup.constraint_code = dim.zone_pattern OR srup.constraint_code = ANY(dim.cross_refs_srup)`.

**Curation rule**: only include cross-refs where (a) the PDM source_text literally names the SRUP layer, OR (b) the PDM uses explicit encompass language ("integrados na X", "tipologias de X", "abrangida por X", "servidão non aedificandi das Redes X e Y"). Conservative — does not include conceptual/definitional-only relationships (e.g. POC-OMG Margem is part of DPH by national law but the PDM doesn't say so).

Zone_patterns that ARE SRUP layer names (`RAN`, `ZPE`, `ZEC`, `IC`, `ARPSI_Floodplain`, `Perigosidade_Incendio_Rural`, `AreasProtegidas`) are omitted from this table — the silver JOIN handles them via direct equality.
