## For future Claude

This is a decision record about pinning `apps/` to SQLAlchemy 1.4 (downgrading from 2.0.36) so the [[2026-05-05-uv-workspace-shape]] unified-resolution can satisfy Airflow 2.10's SQLA<2.0 constraint. Read this when adding SQLAlchemy code to apps/ (you must use the 1.4 API), considering Airflow 3 migration (the constraint disappears), or evaluating workspace-vs-split-projects trade-offs.

## Decision

House4House's workspace pins `sqlalchemy>=1.4,<2` globally. The `apps/` member, which previously had `sqlalchemy==2.0.36` declared, accepts the 1.4 constraint via the workspace's unified resolver.

The concession is workspace-wide because Airflow 2.10's `airflow.providers.postgres.hooks.postgres` (used by every DAG in [[idealista]], [[jll]], [[remax]], [[zome]], and every GIS pipeline) requires SQLA <2.0.

## Why

The workspace shape ([[2026-05-05-uv-workspace-shape]]) means a single root `uv.lock` resolves the dep graph for both apps and pipelines. Two version-incompatible declarations of the same library can't coexist there:

- `pipelines/` (via Airflow 2.10): `sqlalchemy>=1.4,<2`
- `apps/` (originally): `sqlalchemy==2.0.36`

The workspace was the chosen topology for separate reasons (lockfile coherence, Docker-image scoping, cross-member imports). Honoring it forced one of:

(a) Keep apps at SQLA 2 → switch to two independent projects (drops [[2026-05-05-uv-workspace-shape]]).
(b) Keep workspace → downgrade apps to SQLA 1.4.

The Phase 1 spike audited apps/ for SQLAlchemy code: **zero usage was found**. Apps reads from PostgreSQL via `streamlit-postgres` / `psycopg2` directly; no ORM models, no Engine objects, no SQLAlchemy-specific call sites. The downgrade was a paper-only declaration — no code rewriting, no behavior change.

## Options considered

1. **Pin workspace-wide to SQLA 1.4** (chosen) — apps accepts the constraint; zero rewrite cost confirmed by audit.
2. **Split workspace into two independent projects** — rejected because apps's actual SQLA dependency is zero (no benefit from breaking workspace coherence).
3. **Migrate Airflow to 3.x to lift the SQLA <2 constraint** — rejected as out-of-scope (large migration, see [[2026-05-05-cosmos-pin]] for related Airflow-3 concerns).
4. **Use `uv` overrides to force divergent SQLA versions per member** — rejected; uv supports this in principle but it produces hard-to-reason-about resolution and undermines the "one lockfile" promise of the workspace shape.

## Consequences

- Apps must use SQLA 1.4 API if SQLAlchemy ever lands there. The 1.4 ↔ 2.0 differences (`Mapped[T]` typing, `select()` shape, `Session.execute()` boilerplate) matter only when ORM-style code is written.
- A future Airflow 3 migration unblocks SQLA 2.0. When that happens, this concession dissolves naturally — the upper-bound pin is removed.
- The workspace's unified-resolution promise stays intact ([[2026-05-05-uv-workspace-shape]] holds).
- Hard-blocks any plan to bring in a third-party library that requires SQLA 2.x in apps. A new spatial-ORM library, for example, would force a split-workspace re-decision. Not currently anticipated.

## Status

`accepted` — Phase 1 shipped; baseline holds across Phase 2 and Phase 3 PR 1. High confidence; the audit-confirmed-zero-SQLA-usage finding is the load-bearing fact, and an actionable pre-condition (apps stays SQLA-free until either Airflow 3 lands or we explicitly re-decide) is easy to honor.
