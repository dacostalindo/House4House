## For future Claude

This is a decision record about using self-hosted MinIO for the raw object-storage tier (bronze landing zone), rejecting AWS S3 / GCS / equivalent cloud blob stores. It locks in the self-hosted posture for the storage layer paralleling [[2026-05-10-postgis-as-warehouse]]'s warehouse choice. Read this before adding a new bulk-data source, planning a backup, or considering S3 migration.

## Decision

Raw files land in self-hosted **MinIO** (S3-compatible object store), one Docker container on the same Hetzner box as Postgres (see [[infra]]). Convention: `s3://raw/<source>/<version>/<filename>`. Audit copies of every ingest run land here before bronze loads.

## Why

1. **MinIO has the same API as S3.** All ingestion code uses the boto3 / s3fs / dlt-S3 destination libraries, configured to point at the local MinIO endpoint. A future S3 migration is a config-change, not a code-change. This decoupling means the choice is reversible at low cost.
2. **Cost asymmetry compounds across high-volume bulk sources.** [[osm]] is ~1.5 GB / month; [[bgri]] is ~459 MB; [[bupi]] is ~600 MB; [[lidar]] is ~1 GB. ~30 sources × monthly cadences × ~year = several TB of egress + storage. S3 monthly: ~$60-100 at this scale. MinIO on a 4 TB NVMe slot: ~€0.
3. **Latency matters for the audit-copy pattern.** [[bronze-permissive]] mandates that every ingest writes a raw audit copy before transformations run. With MinIO on the same Docker network as Postgres, writes are <1ms; with S3, every audit-copy round-trip is ~50-200ms. Per-DAG that's negligible; aggregated across ~25 daily ingests, it adds up.
4. **No S3-specific feature is essential.** We don't use S3 lifecycle policies (we keep everything), versioning (bronze SCD2 handles versioning per [[scd2-row-hash]]), cross-region replication (single-region operations), or Glacier (no cold-tier archival need yet).

## Options considered

1. **MinIO self-hosted** (chosen).
2. **AWS S3** — same API, ~7-10× cost at our volume. Rejected on cost.
3. **Google Cloud Storage / Azure Blob** — same shape as S3. Rejected for same reasons.
4. **Local filesystem (no object-store layer)** — would simplify infra by one Docker container. Rejected because the S3 API gives us a clean abstraction (audit copy, bulk-list, prefix-glob) that filesystem operations would have to replicate. The S3-shim cost is small; the future-portability benefit is real.
5. **Backblaze B2 / Wasabi (cheap S3-compatible cloud)** — ~$5/TB/month, ~10× cheaper than S3. Rejected because the savings vs MinIO-on-existing-NVMe is small + we'd add a cloud-credentials operations dependency for marginal benefit.

## Consequences

- MinIO console at `:9001`, API at `:9000`. dlt + boto3 + Airflow pipelines use `MINIO_ENDPOINT` + `MINIO_ACCESS_KEY` + `MINIO_SECRET_KEY` Airflow Variables.
- Backups: per-bucket `mc mirror` to a second NVMe slot daily; full server snapshot weekly. No off-site backup yet (acceptable for current data-loss tolerance).
- Disk capacity = NVMe slot capacity. At ~3-5 TB working set growth per year, the 4 TB NVMe slot is good for ~6-12 months. Trigger for upgrade: >80% utilization.
- A future S3 migration is realistic: change endpoint URL + credentials in Airflow Variables, restart pipelines. No code changes anticipated.
- `make install-cron` host-side wiki-lint cron does NOT use MinIO; it writes to `wiki/lint-reports/` on the host filesystem.

## Status

`accepted` — Phase 1 baseline holds; storage layer has not surfaced operational issues. High confidence.
