## For future Claude

This is a decision record about staying on Apache Airflow 2.10 rather than migrating to Airflow 3, with the migration scoped as a future explicit project. Companion to [[2026-05-05-cosmos-pin]] (which pinned Cosmos `<1.7` because 1.7+ imports `airflow.sdk`). Read this when reviewing whether to start an Airflow 3 migration, evaluating a feature that requires Airflow 3, or wondering why the codebase has Airflow-2-flavored imports throughout.

## Decision

Apache Airflow **2.10.4** (matching the `apache/airflow:2.10.4-python3.12` base image) is the orchestrator. Migration to Airflow 3 is **out of scope for current Phases**. Trigger condition for future migration: at least two distinct concrete needs that Airflow 3 resolves cleanly (e.g., native dataset triggers + AsyncIO operators) — not "let's stay current."

## Why

1. **Airflow 3 migration cost is non-trivial.** Major touchpoints: `airflow.providers.postgres.hooks.postgres` patterns used across every DAG, DAG-decorator changes, provider-version bumps, deprecated-API replacements, plus Cosmos's own bump to 1.7+. Conservative estimate: ~1-2 weeks of focused work + a stabilization tail.
2. **Airflow 2.10 is the long-term-supported line.** Apache committed to security backports through 2026; the migration isn't a deadline-driven necessity.
3. **No current feature need requires Airflow 3.** UC-1 / UC-2 / UC-3 + Phases 1-7 of dev-tooling are all expressible in Airflow 2.10 + Cosmos 1.6. Async operators, dataset triggers, taskflow improvements — all "nice to have," none "required."
4. **The deferred-but-acknowledged posture lets us stay coherent.** Pinning explicitly (Cosmos `<1.7`, sqlalchemy `<2`, airflow `2.10.4`) avoids the "stale-but-not-deliberately" anti-pattern. We KNOW we're on 2; we know WHY.

## Options considered

1. **Stay on Airflow 2.10 + Cosmos 1.6** (chosen) — pinned + acknowledged.
2. **Migrate to Airflow 3 now** — rejected; migration cost is real, no concrete trigger.
3. **Replace Airflow with Dagster** — Dagster has cleaner data-asset semantics + better dbt integration. Rejected because (a) we already have Cosmos providing dbt-task-graph generation, (b) Dagster's spatial/scraping ecosystem is thinner, (c) migration cost is way higher than Airflow 2 → 3.
4. **Replace Airflow with Prefect** — Prefect 2.x has nice ergonomics but smaller spatial-tooling ecosystem. Same rejection logic as Dagster.
5. **Replace Airflow with Mage / Kestra / etc.** — too speculative for a production orchestrator. Airflow's batch-tested operator ecosystem (Postgres, S3, GCS, Snowflake, dbt, K8s, etc.) is the moat.

## Consequences

- All DAGs use Airflow-2-flavored imports (`from airflow import DAG`, `from airflow.providers.postgres.hooks.postgres import PostgresHook`, etc.).
- Cosmos pinned `>=1.6,<1.7` (per [[2026-05-05-cosmos-pin]]); the same pin lifts when migration starts.
- SQLAlchemy pinned `<2.0` (per [[2026-05-08-sqla-1.4-concession]]) because Airflow 2 transitive deps require it.
- Airflow 3-only features (improved Iceberg support, async loaders, native dataset triggers) are unavailable. Acceptable trade-off.
- A future migration is a self-contained project: bump Airflow base image, bump Cosmos to 1.7+, lift sqlalchemy pin to `>=2.0`, sweep DAG decorators + import paths + provider versions, run full test suite.

## Status

`accepted` — Phase 1 + Phase 2 + Phase 3 PR 1-5 all shipped on this baseline. Production stability is high; no Airflow 2 EOL pressure on the current horizon.
