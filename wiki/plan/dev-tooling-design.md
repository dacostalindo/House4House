## Problem Statement

House4House is a 21.8k-LOC monorepo (111 Python files, 40 Airflow DAGs, 111 dbt models, Streamlit app, PostGIS warehouse) running production data ingestion for Portuguese real estate and regulatory GIS data. It has zero formal dev tooling: no `pyproject.toml`, no lockfile, no linter, no formatter, no type checker, no CI/CD, no Claude Code context files. Dependencies are split across `apps/requirements.txt` (Streamlit) and raw `pip install` lines in `Dockerfile.airflow` (everything else), with no version coordination between them.

Day-to-day this manifests as: slow Docker rebuilds, dependency drift between local + Airflow image, no protection against style drift across 40+ DAGs, no automated safety net on push, and Claude sessions that re-derive non-obvious project rules every time.

## What Makes This Cool

Two compounding bets. First, the templated patterns across the codebase (GIS sources, bronzeŌåÆstgŌåÆsilver layering, ZenRows two-pass scrape) are exactly the kind of structure where Claude Code skills + CLAUDE.md hierarchy compound rather than decay. Second, every Idealista listing description carries free-text Portuguese prose that today is invisible to the warehouse ŌĆö Pydantic AI turns that prose into typed silver columns (elevator, parking, energy class, view type, condo fee, renovation year), unlocking real estate signals that competitor portal data does not surface.

## Constraints

- **Solo developer** (`REPO_MODE: solo`).
- **Two-container deployment**: Streamlit container ([apps/Dockerfile](apps/Dockerfile)) + Airflow LocalExecutor (`Dockerfile.airflow`, base `apache/airflow:2.10.4-python3.12`). Different base images, different runtime needs.
- **Python 3.12** floor (Airflow base image). Ruff `target-version = "py312"`.
- **dlt 1.25 + dbt-postgres 1.9 + Cosmos** is the production stack. Untouched.
- **Zero existing tests**. Targeted tests added in this plan; broad coverage out of scope.
- **Bronze-is-permissive** is an existing project rule. Pydantic does not enter dlt resources.

## Premises (all confirmed in office hours)

1. Now is the right time for tooling investment.
2. Two-container deployment won't fight uv consolidation **ŌĆö resolved as uv workspace, two members, per eng-review. Phase 1 spike refinement: workspace works only if apps drops to SQLAlchemy 1.4 (Airflow 2.10's transitive constraint); apps has zero direct SQLAlchemy code so the concession costs nothing.**
3. Pydantic on configs is real value, not cargo-cult.
4. Foundation-first ordering is correct.
5. Solo-dev reality justifies the spend.

### Phase 3 ŌĆö Claude Code integration (~2 days, LLM Wiki + /wiki-lint skill + local cron)

**3a. CLAUDE.md hierarchy** (revised per devex-review W4: wiki-only)

Per devex-review W4 (2026-05-08), CLAUDE.md files become thin pointer-layer files. The full content (why/how/history/examples for each rule or pattern) lives in `wiki/concepts/`. CLAUDE.md files contain only the prescriptive one-liner + a link to the wiki page. Single source of truth = wiki.

**Each area-CLAUDE.md follows the same shape (per eng-review A1 + devex-review F2: schema-as-enforcement with file-to-concepts routing):**

```markdown
## Schema for Claude Code (read this first)

Before editing files in this area, **read the wiki/concepts/ page(s)
mapped to your task type below**. Don't read every page on every edit ŌĆö
read the ones relevant to what you're touching. The wiki is the
project's source of truth for rules and patterns; this file routes you
to the right pages.

## Task ŌåÆ concepts routing

- Editing a dlt resource (`source.py`, `*_dlt.py`, `*_dlt_dag.py`):
  read wiki/concepts/pydantic-not-in-dlt.md + bronze-permissive.md.
- Editing a scraper (idealista/jll/remax/zome/etc):
  read wiki/concepts/zenrows-universal-vs-re-api.md +
  payload-cache-lifecycle.md.
- Adding/editing an SCD2 sink:
  read wiki/concepts/scd2-row-hash.md + heartbeat-sidecar.md.
- Editing or adding a config:
  read wiki/concepts/pydantic-not-in-dlt.md (configs are Pydantic;
  dlt resources are not).
- Editing GIS WFS/OGC plumbing:
  read the relevant source page in wiki/sources/ +
  wiki/concepts/medallion-layering.md.
```

The schema header is the **enforcement mechanism** that makes pointer-only safe ŌĆö pointer-following becomes a Claude-Code-level obligation, not a hope. The task-based routing (per devex-review F2) keeps Claude from reading every concept page on every edit while still firing the relevant rules. This is the karpathy pattern applied at area level: the schema (CLAUDE.md) tells the LLM HOW to use the knowledge layer (wiki/concepts/), and the rules in the wiki only fire reliably because the schema requires reading them at the right times.

**Specific area files:**

- Root `CLAUDE.md`: stack overview (3 lines), run-locally one-liner, **explicit pointer to `wiki/CLAUDE.md` for project knowledge schema** (per devex-review W1), schema header (read wiki before editing), link to area-specific CLAUDE.md files.
- `pipelines/CLAUDE.md`: schema header + concept index ŌĆö `bronze-permissive: see wiki/concepts/bronze-permissive.md`, `SCD2 row-hash policy: see wiki/concepts/scd2-row-hash.md`, `Pydantic guardrail: see wiki/concepts/pydantic-not-in-dlt.md`, `ZenRows two-pass: see wiki/concepts/zenrows-universal-vs-re-api.md`, `_payload_cache lifecycle: see wiki/concepts/payload-cache-lifecycle.md`. The full treatment lives in the wiki.
- `dbt/CLAUDE.md`: schema header + index ŌĆö pointers to wiki/concepts/medallion-layering.md, wiki/concepts/staging-yaml-conventions.md.
- `apps/CLAUDE.md`: schema header + index ŌĆö pointers to wiki/concepts/streamlit-keplergl-quirks.md, wiki/concepts/geopandas-postgis-quirks.md.

**Rationale for wiki-only with schema enforcement (per eng-review A1):** avoids the drift risk of "same content in two files" while keeping rule reliability via the schema header. The 30-second-readable quality of CLAUDE.md is preserved because it's now: schema header (~6 lines) + concept index (~5-10 lines).

**3b. `.claude/settings.json`**

**3c. Hooks (`.claude/hooks/`)** ŌĆö revised per eng-review A2 + C2 (failure semantics)

Hook failure policy (per Claude Code spec, `PostToolUse` non-zero exit blocks the message):

- **Advisory (exit 0 with stderr warnings):** ruff findings, ty check (Phase 6+). Style and type warnings inform but don't block ŌĆö Claude can still proceed with the edit.
- **Blocking (non-zero exit):** dbt parse failures (real ref/source/macro errors), destructive Bash guards (PreToolUse).

Hook list:
- `PostToolUse` on Edit/Write of `*.py` ŌåÆ `ruff check --fix` + `ruff format` on the changed file only. **Always exit 0** even when issues remain (advisory).
- `PostToolUse` on Edit/Write of `dbt/models/**/*.sql` ŌåÆ `dbt parse --select <model>+` scoped to the touched model and descendants (~1-2s, vs 5-30s for full parse). **Non-zero exit blocks** when parse fails (real schema error).
- `PreToolUse` blocking destructive Bash on `warehouse/init/` and `airflow db reset` patterns. **Non-zero exit blocks** by design (this is the guardrail).
- After Phase 6: `PostToolUse` on `*.py` ŌåÆ `ty check`. **Always exit 0** (advisory until ty graduates per TODO).

**Schema document (`wiki/CLAUDE.md`)** ŌĆö the load-bearing piece. Tells Claude exactly:

- **Page conventions**: filename rules (kebab-case, lowercase), required sections (`## Source`, `## Schema`, `## Quirks`, `## Last verified` for source pages; `## Decision`, `## Why`, `## Consequences`, `## Status` for decision records), YAML frontmatter (date, tags, source-count).
- **Ingest workflow** (per devex-review W2: triggered both ways):
  - Free-form: user says "ingest raw/article.md" or pastes a URL ŌĆö Claude reads it, writes/updates the relevant wiki page, updates `index.md`, appends to `log.md` in the format `## [YYYY-MM-DD] ingest | <title>`.
  - Skill-routed: `/wiki-ingest <path-or-url>` ŌĆö Phase 7 skill candidate, codified after the workflow has been done 3-5 times manually. Same operations, but reproducible across sessions. (Note: `/wiki-lint` ships in Phase 3e since the launchd cron depends on it from day 1; `/wiki-ingest` waits for Phase 7 since manual ingest works fine in the interim.)
- **Page-discovery during ingest** (per devex-review W7): Claude reads `index.md` first, then `grep`s the wiki for keyword overlap with the new source. Only after that targeted set is identified does Claude open and update specific pages ŌĆö never read all 30+ pages on every ingest.
- **Query workflow**: when answering a question that touches accumulated knowledge, Claude reads `index.md` first, drills into relevant pages, and synthesizes ŌĆö and offers to file the answer back into the wiki if it produced new synthesis worth keeping.
- **Lint workflow** (per devex-review W3 + eng-review A2 + user-pivot to Claude-Code-headless):
  - **Mechanism**: a `/wiki-lint` Claude Code skill (Phase 3e deliverable) invoked headlessly via `claude -p /wiki-lint --max-turns 5`. No Airflow DAG, no direct anthropic SDK use ŌĆö reuses the user's Claude Code subscription instead of per-token billing.
  - **Schedule**: macOS `launchd` plist at `.claude/wiki-lint.launchd.plist`. Install via `make install-cron` (per devex-review F1) ŌĆö Makefile target copies the plist into `~/Library/LaunchAgents/` and runs `launchctl bootstrap gui/$UID ~/Library/LaunchAgents/wiki-lint.plist`. Fires Sunday 06:00 local time. Runs:
    ```bash
    cd <repo> && claude -p /wiki-lint --max-turns 5 \
        > wiki/lint-reports/$(date +%Y-%m-%d-%H%M%S).md 2>&1
    ```
  - **Skill location**: `.claude/skills/wiki-lint/SKILL.md` ŌĆö contains the prompt + workflow conventions. Skill is version-controlled, evolves alongside `wiki/CLAUDE.md` schema.
  - **Cost**: $0 incremental (covered by user's Claude Code subscription).
  - **Failure mode**: if Mac is off, lint doesn't fire that week. Acceptable for advisory linting. User notices "no Monday lint report" as the signal. No retries; if a run fails, next week's cron picks up.
  - **Lint surface**: same as before ŌĆö contradictions between pages, stale claims (`Last verified` > 90 days), orphan pages, concepts mentioned but lacking their own page, missing cross-references, index.md drift.
  - **Output**: skill writes the lint report directly to `wiki/lint-reports/YYYY-MM-DD-HHMMSS.md` (the redirect captures everything Claude writes; the skill is instructed to format output as a coherent markdown report). Skill also appends one summary line to `wiki/log.md` AND updates the `Last lint run: YYYY-MM-DD` line at the top of `wiki/index.md` (per devex-review F5 ŌĆö gives the user a freshness indicator they see passively whenever they open the index).
  - **On-demand alternative**: same skill invoked manually whenever you want ŌĆö `claude -p /wiki-lint` from any session.
- **No content duplication** (per devex-review W4): rules in `pipelines/CLAUDE.md` and the other CLAUDE.md files are pure 1-line pointers to `wiki/concepts/<topic>.md`. The wiki is the single source of truth. CLAUDE.md hierarchy is a thin index.

**Schema amendments before PR 2 ships (per obsidian-second-brain `references/` review, locked 2026-05-08):**

Three small additions to `wiki/CLAUDE.md` schema land as the first commit of PR 2 (before the ~30 generated pages depend on them). All three come from the obsidian-second-brain `references/ai-first-rules.md` review; the rest of the `references/` folder was evaluated and skipped as personal-vault-flavored.

1. **`## For future Claude` preamble** (required section, all page types). 2-3 sentence summary in plain English, immediately after frontmatter, before the schema-required sections. Format: `This note is a [type] about [topic]. It [main purpose]. [Optional caveat about staleness, confidence, or scope.]` The preamble lets a future Claude Code session judge relevance in ~10 seconds before parsing the full page. **Why this fits House4House:** the schema-header routing in CLAUDE.md hierarchy directs Claude to specific wiki pages on every edit. Without a preamble, Claude must read the full page to know whether the routing was a hit or a near-miss. With a preamble, the relevance check is one section. ROI compounds across thousands of edit-loop reads over the project's lifetime.
2. **`confidence: high | medium | speculation` frontmatter field** (decisions/ pages only). Tells a future Claude session how much to trust an old ADR when proposing changes. `high` = battle-tested in a Phase spike or production run; `medium` = picked deliberately but not yet stress-tested; `speculation` = best-guess at decision time, expected to be revisited. Apply ONLY to decisions/ ŌĆö sources/concepts/pipelines have their confidence implicit in the underlying code. **Why scoped narrowly:** putting confidence on every page type adds noise (a `wiki/sources/idealista.md` page is "as confident as the code"; the field is meaningless there). Decisions are different ŌĆö they're judgment calls made under uncertainty, and a future ADR author rewriting one needs the confidence signal to know whether they're overturning a tested choice or refining a guess.
3. **Cross-links are mandatory, using Obsidian-style `[[wikilinks]]`** (rule applies to every page; enforced by `scripts/wiki_health.py` as a BLOCKING CI finding per Phase 4e). Every wiki page MUST link to every canonical wiki page it references on first mention via `[[wikilinks]]`. Resolution rule (documented in `wiki/CLAUDE.md` schema): `[[name]]` resolves to the first match across `wiki/{concepts,sources,decisions,pipelines}/<name>.md` by basename; ambiguity is disambiguated via subdir prefix (`[[concepts/scd2-row-hash]]`); human-readable labels via `[[name|label]]`. Subsequent mentions don't repeat the link. **Why `[[wikilinks]]` not plain markdown:** the user reads the wiki in Obsidian (graph view + backlinks panel + autocomplete on `[[` are killer features); Claude Code reads the syntax as text and resolves via the schema rule. Both modes work. The only UX caveat: Claude Code's built-in VSCode markdown preview doesn't render `[[wikilinks]]` clickable (literal text), but Claude as an LLM still resolves them correctly during reads/edits ŌĆö and the IDE preview is a marginal use case for an LLM-maintained wiki anyway. **Why mandatory, not advisory:** the wiki's value compounds with graph density. A future Claude session (or human in Obsidian) reading `wiki/sources/idealista.md` and seeing "SCD2 + ZenRows two-pass" should get one-click access to the concept pages. That's the difference between a flat collection and a knowledge graph. Mid-PR caveat: forward `[[wikilinks]]` from sources/ (commit 2) to not-yet-created concepts/ pages (commit 3) appear "broken" in intermediate states; the BLOCKING check fires only when a fully merged wiki has missing cross-refs.

These amendments DO NOT bump the `wiki/CLAUDE.md` schema version (still v1) ŌĆö they're additive and the existing PR 1 scaffold pages (just `index.md` skeleton + `log.md` seed entry) get the preamble retrofit in the same PR 2 prep commit.

```makefile
.PHONY: setup verify test up down lint format clean wiki-lint wiki-lint-fast install-cron uninstall-cron

setup:
	uv sync
	uv run pre-commit install
	@echo "Setup complete. Run 'make verify' to smoke-check."

verify:

test: verify

up:
	docker compose up -d airflow-webserver airflow-scheduler postgres minio

down:

lint:
	uv run ruff check

format:
	uv run ruff format

clean:
	docker compose down -v
	rm -rf .pytest_cache .ruff_cache

# Wiki lint ŌĆö two tiers (per devex-review G2):
#   wiki-lint-fast: programmatic, ~seconds, $0, runs in CI + on every wiki/* edit loop.
#   wiki-lint:      LLM-driven, ~minutes, subscription, runs weekly + on final pre-merge.
# The fast tier is what reviewers should run while iterating on PR 3+ wiki/plan/ pages.
# The full tier gates merge, runs in the weekly cron, and catches semantic issues.

wiki-lint-fast:
	uv run python scripts/wiki_health.py --format=text

wiki-lint:
	@mkdir -p wiki/lint-reports
	@TS=$$(date -u +%Y-%m-%dT%H%M%S); \
	  echo "ŌåÆ Running /wiki-lint (output ŌåÆ wiki/lint-reports/$$TS.log)"; \
	  claude -p /wiki-lint --max-turns 30 > "wiki/lint-reports/$$TS.log" 2>&1 || \
	    (echo "wiki-lint failed; see wiki/lint-reports/$$TS.log" >&2; exit 1)
	@echo "wiki-lint complete."

install-cron:
	cp .claude/wiki-lint.launchd.plist ~/Library/LaunchAgents/wiki-lint.plist
	launchctl bootstrap gui/$$UID ~/Library/LaunchAgents/wiki-lint.plist
	@echo "Wiki lint cron installed. Fires Sunday 06:00 local."

uninstall-cron:
	launchctl bootout gui/$$UID/wiki-lint || true
	rm -f ~/Library/LaunchAgents/wiki-lint.plist
	@echo "Wiki lint cron removed."
```

**Two-tier wiki-lint workflow (per devex-review G2):** while iterating on a `wiki/plan/*` PR, run `make wiki-lint-fast` repeatedly ŌĆö it's seconds-fast and catches the mechanical issues (broken wikilinks, missing frontmatter fields, orphans, duplicate filenames) that account for ~80% of lint findings. Reserve `make wiki-lint` (the full LLM run) for the final pre-merge check, where semantic contradictions and concept-emergence signals are worth the 1-2 minute round-trip. The CI pipeline runs the fast tier on every PR; the weekly Sunday cron runs the full tier; humans run either on demand. Same JSON contract on both so downstream tooling sees a consistent shape.

**4c. CI/CD**

**GitHub Actions** (`.github/workflows/ci.yml`):
- Trigger: `pull_request` + `push: branches: [main]`.
- Steps: `uv sync --frozen` ŌåÆ `ruff check --output-format=github` ŌåÆ `ruff format --check` ŌåÆ `pytest tests/configs/ tests/wiki-fixtures/` ŌåÆ `python scripts/wiki_health.py --format=github` ŌåÆ `dbt parse --no-partial-parse`.
- **Inline annotations** (per devex-review P4-A): `--output-format=github` on ruff emits `::error file=X,line=Y::msg` syntax that surfaces failures directly on the PR diff at the offending line. `wiki_health.py --format=github` and the dbt-parse wrapper emit the same format. Failure rendered inline beats clicking through to the action log.
- **Annotation grouping by source** (per devex-review G5, locked 2026-05-08): once both ruff and `wiki_health.py` are wired into CI, a wiki-touching PR can produce a soup of mixed annotations (broken wikilink errors interleaved with ruff style warnings on a tangential `*.py` change). To keep the soup readable, each source prefixes its annotation message with a fixed bracketed tag ŌĆö `::error file=...::[ruff E501] line too long (95 > 88)`, `::error file=...::[wiki-health] orphan page; not linked from index.md or any source page`, `::error file=...::[dbt-parse] ref('stg_foo') not found`. The reviewer can then mentally group by tag in the PR's "Files changed" view (or filter via the GitHub Actions tab's annotation list). Each step also runs inside its own GitHub Actions `group::endgroup` block so the action log itself is sectioned, and the workflow's final step prints a one-line summary per source (`ruff: 2 errors, 0 warnings | wiki-health: 1 error | dbt-parse: ok`) so the failed-checks UI surfaces what category broke without expanding logs.
- After Phase 6: add `ty check` as advisory step (`continue-on-error: true`), prefixed with `[ty]` in the same scheme.

**pre-commit framework** (`.pre-commit-config.yaml`):
- Same checks fire locally before `git commit`. Uses `ruff-pre-commit` and a local hook for scoped `dbt parse`.
- `pre-commit install` is a one-time setup step in the root README.

**First-PR escape hatch**: pre-existing ruff violations (the 282 baseline that survived the autofix sweep) may flag noise on the first CI run. If so, add a baseline-style suppression to `[tool.ruff.lint.per-file-ignores]` for the affected files; reduce the suppressions opportunistically as the codebase evolves.

### Phase 5 ŌĆö Idealista description enrichment via Pydantic AI (~1-2 days, NEW per eng-review)

    parking_spaces: int | None = Field(None, description="...")

**DAG topology**: separate downstream DAG (`idealista_enrichment_dag.py`), nightly schedule, reads new+changed listings from bronze, writes to `silver.idealista_enrichments`. Decoupled from bronze ingestion cadence ŌĆö LLM costs scale with new prose, not with re-scrapes.

**Failure handling**: invalid LLM output triggers Pydantic AI's built-in retry loop (max 2 retries). Retries exhausted ŌåÆ row skipped, logged to dead-letter table, task does not fail. Bronze permissiveness extends to enrichment.

**7e. `/wiki-import-gstack <artifact>`** (~2-3 hours, NEW per user decision 2026-05-08)

Imports a settled gstack artifact (design doc, research artifact, ceo-plan) into the wiki by extracting decisions, concepts, and source-shaped knowledge into the appropriate `wiki/<directory>/` pages. **The gstack Ōåö wiki boundary** (rationale for this skill, locked 2026-05-08):
- *Belongs in gstack* (`~/.gstack/projects/<slug>/`): review logs (`*-reviews.jsonl`), timeline events, skill telemetry, learnings.jsonl, in-flight design/research/ceo-plan artifacts. These are *how we worked* ŌĆö workflow state, useful for the next gstack session, irrelevant to a future Claude Code session asking "what's the SCD2 row-hash policy?"
- *Belongs in wiki* (`wiki/`): decisions that have been made and are now binding (ADRs in `wiki/decisions/`), patterns that recur (concepts in `wiki/concepts/`), source-of-truth descriptions (sources in `wiki/sources/`). These are *the project's compounding knowledge*.
- *The migration moment*: when a gstack artifact stops being in-flight (the design doc's phase ships, the research artifact's borrows are distributed, the ceo-plan's strategy is locked), the **decisions and patterns inside it** flow into the wiki. The original gstack artifact stays in `~/.gstack/` as the historical record of how the decision got made; the wiki page becomes the operational source of truth.
- `/wiki-import-gstack` codifies that migration moment so it's a single command instead of a manual rewrite.

**Workflow:**
1. User runs `/wiki-import-gstack ~/.gstack/projects/dacostalindo-House4House/manuellindo-main-design-20260505-120707.md`.
2. Skill reads the artifact and classifies sections: "Approaches Considered" + "Recommended Approach" ŌåÆ ADR candidate(s); "Risks & Mitigations" ŌåÆ consequences sections of relevant ADRs; "Phase X" sub-sections ŌåÆ typically pattern descriptions for `wiki/concepts/`; "Reusable Scripts" or "Per-Skill Mapping" tables in research artifacts ŌåÆ `wiki/sources/` or `wiki/concepts/`.
3. Skill checks each candidate against the existing wiki: if a `wiki/decisions/<date>-<slug>.md` already exists for the same decision, skill switches to update-mode (proposes a diff), not create-mode. Same for concepts.
4. Skill emits a single proposal block per page: filename, sections, content. User approves/edits per-page (matches PR 2's "review in batches" pattern).
5. On approval, skill writes the pages, regenerates `wiki/index.md`, appends to `wiki/log.md` with `## [YYYY-MM-DD] import-gstack | <artifact-name>` (one log line per import, not per page).
6. Skill **does not delete** the source gstack artifact. The artifact stays in `~/.gstack/` as the history; only its derived knowledge goes into the wiki. (`wiki/decisions/` ADRs may include a `derived_from:` frontmatter field pointing back at the gstack artifact path for traceability.)

**Why structurally different from `/wiki-ingest` (Phase 7 deferred):**
- `/wiki-ingest` takes free-form input (any URL, any docstring, any pasted prose). It must *guess* the structure.
- `/wiki-import-gstack` takes a known gstack artifact shape. It can *parse* the structure (Phase X / Approaches / Risks / Open Questions) deterministically and extract decisions cleanly.
- This means `/wiki-import-gstack` produces higher-quality wiki content with less manual cleanup. Build it before `/wiki-ingest` if both Phase 7 candidates are queued, since gstack artifacts are the most common ingest source for this project.
